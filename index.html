The missing layer

The trillion-dollar prize only materializes if merchants can execute agent intent safely.

Consumer agents can express intent. Merchants still own the systems that make commerce real: catalog, price, inventory, promotions, loyalty, shipping, returns, refunds, risk rules, and brand policy.

Why chatbots failed

Conversation is not execution.

The first wave put probabilistic reasoning on the live customer path. That made commerce slower, more expensive, less predictable, and harder to certify. A customer may tolerate ambiguity during discovery. A merchant cannot tolerate ambiguity when an agent changes an order, approves a return, applies a discount, or commits to a delivery promise.

What changes now

Agents need a controlled merchant endpoint.

Agentic commerce turns shopping from search into delegation. That shifts the burden from answering questions to executing verified commercial intent. The merchant runtime becomes the trusted boundary between open-ended agents and operational systems of record.

System 2 · Offline

Learn, reason, compile.

LLMs and agents discover workflows, analyze failures, simulate edge cases, generate tests, and compile merchant-specific skills outside the production hot path.

System 1 · Online

Verify, route, execute.

Live interactions run through deterministic software: typed interfaces, evidence contracts, policy gates, regression tests, telemetry, rollback, and latency budgets.

Technical architecture

A merchant runtime needs compiler discipline, not prompt discipline.

milli.run keeps System 2 reasoning out of covered production paths. LLMs inspect failures and propose new skills offline. Online execution uses small, testable components with explicit contracts.

Coverage gate

Intent router before execution.

Fast classifiers route only covered workflows into deterministic execution. Low-confidence or novel intents go to fallback, preserving safety without blocking the learning loop.

intent → confidence threshold → covered skill | fallback | human queue
Extraction layer

Structured slots, repaired against the catalog.

spaCy extraction is paired with deterministic repair logic: exact catalog-token preservation, longest-match attribute scans, size and price parsing, and merchant-specific lexicons.

spaCy NER → repair_slots() → color/size/price/attribute constraints
Evidence layer

Claims must resolve to records.

Responses and actions are checked against catalog, order, policy, customer, and fulfillment state. The runtime is designed to prevent unsupported claims from reaching the customer or agent.

claim → typed repo call → record match → policy gate → response/action
Compiler loop

Fallbacks become tests before skills.

Offline agents analyze failures, synthesize edge cases, generate regression fixtures, and propose workflow skills. Deployment happens only after shadow evaluation and versioned rollout gates.

fallback trace → LLM analysis → test fixture → skill candidate → shadow A/B → rollout

Why we are building this

The team has seen commercial intent break from every side of the stack.

The founding group — K.R., W.Z., D.H., A.D., and D.S. — brings operating history across a recently exited startup, enterprise customer service AI, global consumer brands, demand generation, and developer platforms.

Pattern observed

Intent is valuable, but rarely executable.

Brands often infer intent through indirect media and data systems. Support tools talk around the systems that actually hold catalog, order, policy, and fulfillment truth. Agents can express what should happen, but merchants still need a runtime that decides what may happen.

Operating lesson

Reasoning needs controls.

Commerce scales when probabilistic intelligence is wrapped in deterministic execution, evidence contracts, policy checks, observability, and rollback. milli.run exists to make that control layer available to merchants before agents become the dominant commerce interface.

Deployment path

Start with support workflows. Expand into agentic commerce execution.

Support is the natural wedge because it already contains repeated commercial intent: where is my order, can I return this, which variant fits, can I exchange it, is this eligible, what can you recommend.

Phase 1

Ingest

Sync tickets, macros, policies, product catalog, order states, returns, and fulfillment data. Establish baseline coverage and evidence requirements.

Phase 2

Compile

Train routers and extractors, compile covered workflows, generate regression tests, and run in shadow mode against real support sessions.

Phase 3

Route

Route covered traffic through deterministic execution paths. Escalate low-confidence interactions and feed fallbacks back into the compiler loop.

Architecture thesis

LLMs learn offline. Deterministic merchant software executes online.

milli.run turns agentic commerce from runtime reasoning into compiled execution: fast enough for customer support, reliable enough for merchant operations, and controlled enough for brands and retailers to trust.
