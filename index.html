Architecture

The compiler paradigm shift.

Standard RAG shopping assistants pay both retrieval and generation cost on every query. The compiled path extracts structured slots locally, repairs catalog-specific values deterministically, then sends normalized constraints to the same product ranker.

1. Same Ranker, Isolated Variable
The benchmark isolates slot extraction. Golden labels, spacy+repair, and the LLM extractor all feed the same search_products() ranker, so the result measures extraction quality rather than downstream ranker differences.
2. Catalog-Specific Repair Beats Normalization
Product catalogs contain vendor-specific values such as 152-green. The compiled path preserves exact color codes through lexicon and regex repair, while general-purpose LLM extraction can paraphrase values into plausible but wrong normalized colors.
3. Fast Path With LLM Fallback
Covered intents execute with zero hot-path LLM calls. If classifier confidence falls below the default 0.70 threshold, the system routes to the LLM fallback rather than pretending deterministic coverage exists.

Interpretation: vector RAG reaching 95% confirms the previous keyword-style baseline was too weak. A capable LLM with semantic retrieval can approach compiled-path accuracy, but only with roughly 16 seconds of latency and a full LLM generation per query. The remaining 5-point gap traces to variant disambiguation, especially exact vendor color codes such as 152-green.
Methodology note: earlier random 80/20 splits allowed paraphrases from the same seed utterance to appear in both train and test. The v3 model uses seed-grouped splitting in compiler/train_spacy.py, and generate_synthetic_data.py now writes seed_id to every row. Old CSVs without seed_id fall back to random split with a printed warning.
Strategy

Why these benchmark results matter.

The v3 result supports a narrower and stronger wedge: compiled extraction can match oracle accuracy for covered catalog workflows while avoiding the latency and token cost of runtime LLM reasoning.

Why is this big?
Covered Workflow Wedge
Product retrieval and variant disambiguation are high-frequency commerce tasks where structured constraints matter. The initial wedge is not all support; it is covered workflows with measurable slot, ranker, and evidence accuracy.
Why now?
Latency + Cost Pressure
The LLM extractor reaches 95% ASIN match but takes 4,245ms. Vector RAG also reaches 95% but takes 15,821ms. The compiled path reaches 100% at 4ms extraction latency and no LLM token cost on covered intents.
Why will it last?
Seed-Grouped Generalization
Every fallback trace can become a regression test case. The software library grows with usage when review, compilation, and deployment gates are enforced.
Why us?
Variant-Level Accuracy
The key failure mode is exact variant disambiguation. Semantic retrieval can find the product family, but catalog-specific slot repair is better suited to exact color, size, price, and attribute constraints.
Implementation

Pilot Installation Process

A low-risk, phase-by-phase onboarding roadmap to transition customer service operations to a compiled, deterministic runtime while maintaining safety gates and agent fallbacks.

Phase 1
Ingest & Train
We sync historical tickets, macros, policies, product catalog data, and order-state examples, establishing baseline coverage and training custom intent classifiers offline.
Phase 2
Compile & Verify
Core workflows are compiled into Elixir skills and integrated with your live APIs, running in shadow mode to audit accuracy against real agents.
Phase 3
Route & Scale
Routine tickets are routed gradually with automated verifier checks. Low-confidence interactions route to standard helpdesk queues.
W1–W2
Catalog Sync
Sync product catalog, order-state fixtures, policies, and active macros. Deploy shadow monitoring to baseline customer intents.
W3–W4
Extraction Training
Train FastText intent classifiers and spaCy extractors on historical tickets; establish confidence thresholds.
W5–W6
Skill Compiling
Compile core skills (WISMO tracking, return windows) into Elixir; connect live carrier and Shopify APIs.
W7–W8
Shadow A/B Run
Run milli.run in shadow mode. Verify product/variant match targets, evidence pass rates, escalation behavior, and latency against live agent sessions.
W9–W10
Controlled Launch
Route 5% of routine incoming tickets to milli.run. Enable automated verifier audits and escalation to the existing helpdesk queue.
W11–W12
Production Scale
Scale to 50% traffic. Stream shadow fallbacks to compiler to automatically expand intent skill library.

Autonomic support compiled to code. Not prompts.

By using LLMs offline for compilation, evaluation, and fallback, milli.run targets sub-50ms execution, evidence-checked outputs, and no hot-path LLM token cost on covered workflows.
